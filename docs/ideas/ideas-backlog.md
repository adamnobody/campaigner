# Ideas Backlog

> Парковка идей для развития Campaigner после завершения миграции на Rust/Tauri.
>
> **Этот документ не roadmap.** Это отфильтрованный список идей, в которые мы верим как в потенциально ценные. Включение идеи сюда **не означает обязательство её реализовать** — это значит, что идея прошла базовый фильтр на адекватность и заслуживает дальнейшего обсуждения.
>
> Roadmap формируется отдельно из этого backlog после того, как:
>
> 1. Миграция на Rust/Tauri завершена и стабильна.
> 2. Появились первые реальные пользователи (или хотя бы регулярное использование автором).
> 3. Появилось понимание реальных болей, а не воображаемых.

---

## Правила работы с этим документом

1. **Идеи должны быть зрелыми.** Не «было бы прикольно», а «решает конкретную проблему».
2. **Каждая идея описывается по единому шаблону** (см. ниже). Если не можешь заполнить шаблон — идея ещё не готова.
3. **Дубликаты и микро-идеи не добавляем.** Если новая мысль — это уточнение существующей идеи, добавляй её внутрь существующего блока, не плодя новые.
4. **Удаление допустимо.** Если идея со временем потеряла актуальность — переноси в раздел `Архив отказа` с пояснением, почему отказались.
5. **Связи важнее отдельных идей.** Если идея А — фундамент для Б и В, это явно указывается в `Зависимости`.

## Слой 1: Foundation (архитектурный фундамент)

Эти идеи — основа, на которой строятся все остальные. Их реализация должна быть первой в постмиграционном roadmap.

### 1. Canvas — новая основная рабочая поверхность

**Тип:** Foundation  
**Статус:** Validated  
**Сложность:** XL  
**Ценность:** Высокая

**Проблема, которую решает:**  
Текущая карта — это `<img>` подложка + SVG-слой территорий + DOM-маркеры. Эта реализация ограничивает добавление новых типов визуальных объектов: невозможно нормально рисовать линии, кривые, тексты, примитивы, слои. Каждая новая визуальная фича требует отдельного хака.

**Решение:**  
Перенести карту на единый Canvas (или WebGL/SVG-движок; выбор технологии — отдельный ADR), который становится универсальной рабочей поверхностью для любых визуальных объектов: маркеров, регионов, линий, текста, изображений, связей.

**Модель данных / архитектура:**

- Единая система объектов на холсте с общим интерфейсом: `{ id, type, transform, layer, properties }`.
- Слои с независимой видимостью и порядком отрисовки.
- Координатная система с пан/зум/поворотом.
- Сериализация состояния холста в БД (вероятно, через polymorphic-таблицу `canvas_objects` с типом).

**Зависимости:**

- Завершённая миграция на Rust/Tauri.
- ADR по выбору технологии рендеринга (Canvas 2D / WebGL / SVG / гибрид).

**Что блокирует / открывает:**

- **Открывает:** примитивы на карте (идея 2), label transform/curve (идея 3), визуальные связи между объектами, многослойные карты, freeform-надписи, рисование, любые будущие визуальные домены.
- Без Canvas все эти фичи невозможны или потребуют дублирующих реализаций.

**Риски и подводные камни:**

- Большая фича, легко превратить в год работы. Нужен жёсткий MVP-скоуп.
- Производительность: при тысячах объектов важно правильно выбрать технологию рендера и стратегию виртуализации.

**Альтернативы:**  
Развивать текущий SVG-подход — но это упирается в потолок производительности и сложности кода при первой же серьёзной фиче (свободные тексты, кривые).

### 2. Примитивы для добавления на карту

**Тип:** Foundation  
**Статус:** Validated  
**Сложность:** L  
**Ценность:** Высокая

**Проблема, которую решает:**  
Сейчас на карту можно положить только маркер (точка) и территорию (полигон). Невозможно нарисовать линию (граница, торговый путь, река), прямоугольник (область интереса), свободный текст-надпись, иконку, изображение-аппликацию. Это резко ограничивает возможности картографирования.

**Решение:**  
Базовый набор визуальных примитивов как объектов Canvas:

| Тип | Назначение |
|-----|------------|
| `point` | Точка / якорь |
| `marker` | Иконка с привязанной сущностью (расширение существующего) |
| `region` | Полигон с заливкой (расширение существующего) |
| `line` | Линия / полилиния |
| `polygon` | Произвольный многоугольник без заливки |
| `label` | Свободная надпись |
| `icon` | Иконка без сущности |
| `image` | Изображение-аппликация |

**Модель данных / архитектура:**

- Полиморфная таблица `canvas_objects` с `type` и `properties_json`.
- Каждый примитив имеет общие поля (`transform`, `layer`, `locked`, `visible`) и специфичные (точки полилинии, цвет заливки, текст и т.д.).

**Зависимости:**

- Canvas (идея 1) — обязательное условие.

**Что блокирует / открывает:**

- **Открывает:** визуальные карты религий, торговые пути, политические границы, любые ad-hoc обозначения.
- **Открывает:** использование примитивов для отображения доменных сущностей (религии на карте, владения династий).

**Риски и подводные камни:**

- Перегрузка UI: 8 типов примитивов сразу — много. Начать с 4–5 базовых (`line`, `label`, `polygon`, `image`), остальное добавлять по запросу.
- Стилизация: нужно решить, насколько глубоко идти в графический редактор (нельзя стать Figma).

**Альтернативы:**  
Делать каждый примитив как отдельный домен — через год получим 8 несовместимых реализаций.

### 3. Label transform / curve

**Тип:** Foundation  
**Статус:** Validated  
**Сложность:** M  
**Ценность:** Высокая

**Проблема, которую решает:**  
Текстовые подписи на карте сейчас плоские, без поворота и трансформаций. Невозможно подписать реку вдоль её русла, повернуть название региона, выровнять подпись по кривой. Это критично для атмосферной картографии (как в стиле фэнтези-карт).

**Решение:**  
Расширенные возможности для текстовых объектов:

- Rotate (поворот вокруг якоря).
- Scale (масштаб независимый от карты или зависимый).
- Anchor / alignment.
- Curve / path text (текст вдоль произвольной кривой).
- Font size, style, weight.
- Snapping к другим объектам.
- Layer assignment.

**Модель данных / архитектура:**  
Расширение свойств `label` примитива:

- `transform`: `{ rotation, scale, anchorX, anchorY }`
- `pathId`: `Option<CanvasObjectId>` — если задано, текст рендерится вдоль этого пути.
- `style`: `{ fontFamily, fontSize, weight, color, outline }`

**Зависимости:**

- Canvas (идея 1).
- Примитивы (идея 2) — нужны `line`/`path` для curve text.

**Что блокирует / открывает:**

- **Открывает:** атмосферные карты в стиле классических фэнтези-атласов.
- Сильно повышает воспринимаемое качество приложения.

**Риски и подводные камни:**

- Curve text — нетривиальная задача рендеринга, особенно с правильным распределением букв по длине пути.
- Кросс-платформенные шрифты: на разных ОС шрифты выглядят по-разному. Нужно решение для embedded fonts.

**Альтернативы:**  
Оставить плоские подписи — но это сильно режет ценность Canvas как такового.

### 4. Sidebar 2.0

**Тип:** UX  
**Статус:** Draft  
**Сложность:** L  
**Ценность:** Высокая

**Проблема, которую решает:**  
Текущий sidebar — плоский список разделов проекта. С появлением Canvas, объектов на карте и многослойности нужна более мощная панель управления, которая адаптируется к контексту: свойства выбранного объекта, дерево сущностей, слои, связи.

**Решение:**  
Контекстная многорежимная панель:

| Режим | Назначение |
|-------|------------|
| Properties | Свойства выбранного объекта (на холсте или в сущности) |
| Tree | Иерархическое дерево сущностей проекта |
| Layers | Управление слоями Canvas |
| Relations | Связи выбранного объекта |
| Navigation | Текущий sidebar как fallback |

Переключение между режимами — через табы или сегменты.

**Модель данных / архитектура:**  
Чистая frontend-фича, не требует изменений в БД. Зависит от внутреннего state-менеджмента (`selection`, `context`).

**Зависимости:**

- Canvas (идея 1) — sidebar должен управлять Canvas-объектами.
- Примитивы (идея 2) — properties mode должен знать о типах объектов.

**Что блокирует / открывает:**

- **Открывает:** эффективную работу с большим числом объектов на холсте.
- **Открывает:** быстрое редактирование без переходов на отдельные страницы.

**Риски и подводные камни:**

- Перегрузка панели: легко скатиться в «всё во всём». Нужны чёткие режимы и переключения.
- Состояние выделения: что считается «выбранным» — один объект, группа, ничего? Нужна продуманная модель selection.

**Альтернативы:**  
Оставить текущий sidebar и делать отдельные панели для каждого нового контекста — путь к фрагментации UX.

## Слой 2: Worldbuilding улучшения (после фундамента)

Эти идеи усиливают существующие домены и используют фундамент (Canvas, примитивы).

### 5. Шаблоны заметок

**Тип:** UX  
**Статус:** Draft  
**Сложность:** S  
**Ценность:** Высокая

**Проблема, которую решает:**  
Создавая нового NPC, локацию или организацию, пользователь каждый раз набирает одну и ту же структуру заметки вручную. Это замедляет worldbuilding и приводит к несогласованности.

**Решение:**  
Пресеты заметок: пользователь определяет шаблон один раз, при создании новой заметки выбирает шаблон и получает преднастроенную структуру.

**Модель данных / архитектура:**

- Таблица `note_templates`: `id`, `project_id`, `name`, `content_template`, `default_tags`.
- При создании заметки — копирование `content_template` в `content`.

**Зависимости:**  
Никаких.

**Что блокирует / открывает:**  
**Открывает:** более структурированный worldbuilding, легче поддерживать единый стиль записей.

**Риски и подводные камни:**

- Соблазн усложнить шаблоны (переменные, генераторы) — на старте делаем простые статичные шаблоны.

**Альтернативы:**  
Использовать копирование существующей заметки — менее удобно и не масштабируется.

### 6. Иерархия тегов

**Тип:** UX  
**Статус:** Draft  
**Сложность:** M  
**Ценность:** Средняя

**Проблема, которую решает:**  
Текущие теги плоские. При большом проекте (100+ тегов) сложно ориентироваться. Невозможно сгруппировать теги типа `region:north`, `region:south`, `region:east` в одну группу.

**Решение:**  
Древовидная структура тегов: тег может иметь родителя. Фильтрация по родительскому тегу автоматически включает дочерние.

**Модель данных / архитектура:**

- Добавить `parent_tag_id` в таблицу `tags`.
- Логика наследования при поиске/фильтрации.

**Зависимости:**  
Никаких.

**Что блокирует / открывает:**  
**Открывает:** масштабирование тегов на большие проекты.

**Риски и подводные камни:**

- Циклические зависимости (тег как родитель самого себя через цепочку) — нужна валидация.
- Migration существующих плоских тегов — должна быть бесшовной.

**Альтернативы:**  
Использовать префиксы в именах (`region/north`) — работает, но не даёт нормального UI и фильтрации.

### 7. Backlinks в заметках

**Тип:** UX  
**Статус:** Draft  
**Сложность:** M  
**Ценность:** Высокая

**Проблема, которую решает:**  
В worldbuilding критично знать «кто ссылается на эту сущность». Например, открывая заметку о городе, хочется видеть все NPC, события и организации, которые на него ссылаются. Сейчас это требует ручного поиска.

**Решение:**  
Автоматический сбор и отображение обратных ссылок: для каждой сущности — список других сущностей, которые на неё ссылаются (через wiki-links или упоминания).

**Модель данных / архитектура:**

- Расширение существующей системы `wiki_links` с обратным индексом.
- Компонент на странице сущности: «На меня ссылаются: …».

**Зависимости:**

- Существующая система wiki-links.

**Что блокирует / открывает:**  
**Открывает:** навигацию по миру в стиле Obsidian, понимание роли сущности в проекте.

**Риски и подводные камни:**

- Производительность при тысячах ссылок — нужен индекс.

**Альтернативы:**  
Ручной поиск — медленно и ненадёжно.

### 8. Версионирование заметок

**Тип:** Tooling  
**Статус:** Draft  
**Сложность:** M  
**Ценность:** Средняя

**Проблема, которую решает:**  
Worldbuilding — итеративный процесс. Бывает, что хочется откатить изменения или посмотреть, как сущность выглядела месяц назад. Сейчас это невозможно.

**Решение:**  
История правок для заметок (и, опционально, других ключевых сущностей): хранение снимков при значимых изменениях, возможность просмотра и отката.

**Модель данных / архитектура:**

- Таблица `note_revisions`: `id`, `note_id`, `content`, `created_at`, `change_summary`.
- Стратегия снимков: при каждом сохранении / по интервалу / только при ручной отметке (отдельный ADR).

**Зависимости:**  
Никаких.

**Что блокирует / открывает:**  
**Открывает:** безопасный итеративный worldbuilding без страха «сломать всё».

**Риски и подводные камни:**

- Раздувание БД: нужна стратегия очистки старых ревизий.
- Что версионировать кроме заметок? Не делать сразу для всего — начать с `notes`.

**Альтернативы:**  
Полагаться на export/import как ручной backup — не покрывает быстрые откаты.

## Слой 3: Новые домены

### 9. Принципы династий (расширенная модель)

**Тип:** Domain  
**Статус:** Validated  
**Сложность:** L  
**Ценность:** Высокая

**Проблема, которую решает:**  
Текущая модель династий упрощённая: персонажи, члены, семейные связи, события. Не покрывает браки как отдельные сущности, наследование титулов, периоды правления, связи с территориями.

**Решение:**  
Расширенная доменная модель династий:

- **Персонажи** — существующие.
- **Дома / династии** — существующие, с расширенными атрибутами.
- **Родственные связи** — типизированные (отец, мать, супруг, ребёнок, побратим и т.д.).
- **Браки** — отдельная сущность с датами, статусом (действующий / расторгнутый / смерть супруга), детьми.
- **Наследование** — правила (примогенитура, выбор и т.д.), очередь наследников.
- **Титулы** — отдельная сущность с историей владельцев и периодами правления.
- **Периоды правления** — кто, что, когда, какие события.
- **События династии** — расширение существующих.
- **Связи с территориями / городами / религиями** — через общую систему связей.

**Модель данных / архитектура:**  
Существенное расширение существующих таблиц `dynasty_*`. Возможно, выделение титулов в отдельный домен.

**Зависимости:**

- Существующий домен `dynasties`.
- Желательно: общая система типизированных связей между доменами.

**Что блокирует / открывает:**  
**Открывает:** глубокое моделирование политической истории мира, генеалогий, наследственных конфликтов.

**Риски и подводные камни:**

- Сложность UI: легко перегрузить пользователя обилием полей. Нужна продуманная подача (вкладки, прогрессивное раскрытие).
- Migration существующих династий должна быть бесшовной.

**Альтернативы:**  
Описывать всё в свободных заметках — работает, но теряется структурность и связи.

### 10. Религии

**Тип:** Domain  
**Статус:** Validated  
**Сложность:** L  
**Ценность:** Высокая

**Проблема, которую решает:**  
Религии — одна из ключевых сущностей фэнтези-мира, наравне с фракциями и династиями. Сейчас их приходится моделировать через фракции или заметки, что теряет специфику (пантеон, святые места, догматы).

**Решение:**  
Отдельный домен религий с моделью:

- Название, символ, описание.
- Догматы и принципы (через существующий домен `dogmas` с привязкой).
- Святые места (через примитивы на карте + связь с локациями).
- Регионы распространения (territory на карте).
- Иерархия духовенства.
- Отношения с другими религиями.
- Связи с культурами / династиями / государствами.
- Отображение на карте (через Canvas + примитивы).

**Модель данных / архитектура:**

- Таблица `religions` с базовыми полями.
- Связи через `religion_dogmas`, `religion_locations`, `religion_relations`.
- Использование Canvas-примитивов для отображения на карте.

**Зависимости:**

- Существующий домен `dogmas` (опционально, для догматов).
- Canvas (идея 1) — для отображения на карте.
- Примитивы (идея 2) — для регионов и святых мест.
- Желательно: домен «локации» как самостоятельная сущность.

**Что блокирует / открывает:**  
**Открывает:** моделирование религиозных конфликтов, реформаций, миссионерства.

**Риски и подводные камни:**

- Пересечение с фракциями: где граница между «религия» и «религиозная фракция»? Нужен ADR.
- Глубина моделирования: легко уйти в крайность с детализацией.

**Альтернативы:**  
Использовать `factions` с типом `religion` — теряется специфика (нет пантеона, догматов как структурированных сущностей).

### 11. Items / Inventory

**Тип:** Domain  
**Статус:** Draft  
**Сложность:** L  
**Ценность:** Высокая

**Проблема, которую решает:**  
В мире есть значимые предметы: артефакты, реликвии, оружие с историей. Сейчас их можно описать только в заметках — нет структурированной модели «у кого находится сейчас», «кто владел раньше», «какие свойства».

**Решение:**  
Домен Items с базовыми атрибутами и историей владения:

- Название, описание, тип, редкость.
- История владельцев (chain of custody).
- Текущее местоположение (у персонажа / в локации / утерян).
- Свойства (произвольные key-value или структурированные).
- Связи с другими сущностями.

**Модель данных / архитектура:**

- Таблица `items`.
- Таблица `item_ownership_history`.
- Связь с `characters` / `locations`.

**Зависимости:**

- Желательно: домен «локации» для размещения предметов.

**Что блокирует / открывает:**  
**Открывает:** моделирование артефактов и легендарных предметов как полноценных сюжетных элементов.

**Риски и подводные камни:**

- Скольжение в сторону RPG-инвентаря (вес, экипировка, статы) — не наш кейс. Фокус на «значимые предметы», а не «весь шмот».

**Альтернативы:**  
Описание в заметках — работает для уникальных предметов, но без структуры и истории владения.

### 12. Locations (универсальная сущность места)

**Тип:** Domain  
**Статус:** Draft  
**Сложность:** L  
**Ценность:** Высокая

**Проблема, которую решает:**  
Сейчас «место» в Campaigner — либо маркер на карте, либо упоминание в заметке. Нет полноценной сущности «локация» с описанием, NPC, квестами, иерархией (город → район → таверна).

**Решение:**  
Домен Locations:

- Иерархическая структура (`location` может иметь `parent_location`).
- Тип (город, деревня, замок, подземелье, таверна, храм, …).
- Описание, изображение.
- Связи с NPC, фракциями, событиями.
- Привязка к маркеру на карте (опционально).

**Модель данных / архитектура:**

- Таблица `locations` с `parent_location_id`.
- Связи через polymorphic associations (как теги).
- Опциональная привязка к canvas-объекту на карте.

**Зависимости:**

- Желательно: Canvas для красивой привязки к карте.

**Что блокирует / открывает:**  
**Открывает:** моделирование городов с районами, подземелий с комнатами, любой пространственной иерархии.

**Риски и подводные камни:**

- Слишком общая абстракция: «локация» включает и империю, и таверну. Возможно, нужны разные подтипы с разными полями.

**Альтернативы:**  
Использовать существующие маркеры + заметки — теряется иерархия и структурность.

### 13. Bestiary (бестиарий существ)

**Тип:** Domain  
**Статус:** Draft  
**Сложность:** M  
**Ценность:** Высокая

**Проблема, которую решает:**  
В мире есть значимые существа: уникальные монстры, виды, мифические создания. Их моделирование через `characters` неудобно — у них другая природа (нет личности у вида в целом, есть тактика боя, экология, ареал обитания).

**Решение:**  
Отдельный домен Bestiary с фокусом на виды и значимых уникальных существ:

- Название, описание, тип.
- Тактика / поведение.
- Экология (среда обитания, рацион, размножение).
- Ареал распространения (через территории на карте).
- Связи с фракциями (служат, враждуют).
- Различие между «вид» и «конкретное существо» (флаг `is_unique`).

**Модель данных / архитектура:**

- Таблица `bestiary_entries`.
- Связи через теги и polymorphic relations.
- Опциональная связь с canvas-объектами для ареала.

**Зависимости:**  
Никаких критичных.

**Что блокирует / открывает:**  
**Открывает:** моделирование экосистемы мира, тематических подземелий, мифологии.

**Риски и подводные камни:**

- Граница с `characters`: уникальные существа (древний дракон Х) могут быть и персонажами. Нужен ADR о разделении или объединении.

**Альтернативы:**  
Использовать `characters` с тегом `monster` — работает, но теряется специфика (экология, тактика).

### 14. Кастомные календарные системы

**Тип:** Domain  
**Статус:** Draft  
**Сложность:** L  
**Ценность:** Средняя

**Проблема, которую решает:**  
Текущий таймлайн использует, по сути, реальное время. Фэнтези-мир редко использует григорианский календарь — там свои месяцы, недели, эры, фазы лун. Привязка событий к реальным датам ломает погружение.

**Решение:**  
Возможность определить пользовательский календарь для проекта:

- Количество месяцев, их названия и длительность.
- Длина недели и названия дней.
- Эры (с переключением летоисчисления).
- Опционально: фазы лун, сезоны как производные.

Все события в проекте используют этот календарь для отображения дат.

**Модель данных / архитектура:**

- Таблица `calendar_systems` (один на проект, но потенциально несколько).
- Хранение дат в виде «дни от эпохи проекта» с конвертацией в кастомный формат.
- **Архитектурно ломающее изменение:** все существующие даты в проекте требуют миграции.

**Зависимости:**  
Никаких внешних, но требует продуманного ADR.

**Что блокирует / открывает:**  
**Открывает:** атмосферный таймлайн, фазы лун для магии/событий, реалистичные периоды правления династий.

**Риски и подводные камни:**

- Архитектурная ломкость: трогает все домены, где есть даты (`timeline`, `events`, dynasty events, …).
- Migration существующих проектов с реальными датами — отдельная проблема.
- UI ввода дат должен поддерживать кастомный календарь.

**Альтернативы:**  
Оставить реальные даты и переименовывать в UI — теряется реалистичность месяцев и недель.

### 15. Многоуровневые / многослойные карты

**Тип:** Domain (расширение Maps)  
**Статус:** Draft  
**Сложность:** L  
**Ценность:** Высокая

**Проблема, которую решает:**  
Сейчас карта плоская. Невозможно нормально работать с многоэтажными замками, подземельями, городами с уровнями (поверхность / канализация). Невозможно делать переключаемые «политическая / торговая / военная» проекции одной местности.

**Решение:**  
Две связанные концепции:

- **Уровни (levels)** — отдельные карты, связанные через порталы/лестницы (этажи замка, уровни подземелья).
- **Слои (layers)** — overlay над одной картой (политический слой, торговые пути, климат), переключаемые.

**Модель данных / архитектура:**

- Расширение домена `maps`: `parent_map_id`, `level_index`.
- Слои реализуются через атрибут `layer` объектов на canvas (см. идея 1).
- Порталы между уровнями — особый тип canvas-объекта с ссылкой на другую карту.

**Зависимости:**

- Canvas (идея 1) — без него многослойность невозможна нормально реализовать.

**Что блокирует / открывает:**  
**Открывает:** моделирование сложных пространственных структур (городов, подземелий, многомерных миров).

**Риски и подводные камни:**

- UX навигации между уровнями: легко запутаться. Нужен mini-map или breadcrumb.

**Альтернативы:**  
Использовать отдельные карты без связи между собой — работает, но теряется ощущение единого пространства.

## Слой 4: Campaign Mode (опционально, после worldbuilding-этапа)

Отдельная стратегическая ветка: превращение worldbuilding-инструмента в инструмент проведения сессий. Решение о реализации — после того, как worldbuilding-фундамент стабилен.

### 16. Campaign Mode MVP

**Тип:** Foundation (для нового режима)  
**Статус:** Draft  
**Сложность:** XL  
**Ценность:** Высокая (если есть пользователи-ГМы)

**Проблема, которую решает:**  
Campaigner — отличный инструмент для подготовки мира, но не для проведения игры. ГМу во время сессии нужны дайс-роллер, журнал сессии, трекер инициативы, окно для игроков. Сейчас это в других приложениях, переключение ломает фокус.

**Решение:**  
Глобальный режим Play Mode с компактным набором инструментов:

1. Mode switcher (Build ↔ Play).
2. Session Log — журнал сессий (новый домен).
3. Dice Roller с формулами и историей.
4. Quest Log — отдельный домен квестов.
5. Initiative Tracker.
6. Player View — второе окно для показа игрокам (без спойлеров).

**Модель данных / архитектура:**

- Новые домены: `sessions`, `quests`.
- Mode-state глобальный, влияет на UI всего приложения.
- Player View — отдельное Tauri window с подмножеством state.

**Зависимости:**

- Стабильная миграция на Rust/Tauri (для multi-window).
- Желательно: домен Locations (для Quest Log связей).

**Что блокирует / открывает:**

- **Открывает** новый рынок: ГМы, ищущие all-in-one tool для подготовки + проведения.
- **Открывает:** Fog of War, Combat Timers, Random Tables, Player Handouts (все требуют Play Mode).

**Риски и подводные камни:**

- Расширение скоупа: Play Mode легко превратить в Roll20-конкурента. Нужно жёсткое ограничение MVP.
- Concurrent state двух окон: нетривиальная синхронизация. Нужен ADR.
- Решение «нужно ли это вообще» зависит от реальной аудитории. Проверить через user research до начала.

**Альтернативы:**  
Оставить Campaigner как pure worldbuilding tool. Это легитимный выбор — не каждый продукт должен расти вширь.

---

### 17. Fog of War + Player Handouts

**Тип:** Domain (расширение Campaign Mode)  
**Статус:** Draft  
**Сложность:** M  
**Ценность:** Высокая (если Campaign Mode реализован)

**Проблема, которую решает:**  
Игроки не должны видеть всю карту с самого начала. ГМ хочет раскрывать территории по мере исследования и показывать игрокам конкретные изображения/тексты в нужный момент (handouts).

**Решение:**

- **Fog of War** на canvas-объектах: каждый объект имеет флаг `visible_to_players`. В Player View скрытые объекты не показываются.
- **Handouts:** ГМ может «отправить» изображение или текст в Player View одним кликом.

**Модель данных / архитектура:**

- Поле `visible_to_players` (или `revelation_state`) на canvas-объектах.
- Handouts queue — список того, что показывается игрокам прямо сейчас.

**Зависимости:**

- Campaign Mode MVP (идея 16).
- Canvas (идея 1).

**Что блокирует / открывает:**  
Закрывает один из главных пробелов любого VTT-like инструмента.

**Риски и подводные камни:**

- Сценарии разоблачения: одиночные маркеры / целые регионы / зоны. Нужна продуманная модель раскрытия.

**Альтернативы:**  
Использовать отдельные сохранённые скриншоты карт для разных стадий исследования — крайне неудобно.

## Архив отказа

Идеи, которые рассматривались и **сознательно отклонены**. Хранятся здесь, чтобы не возвращаться к ним без новых аргументов.

### ❌ Player Portal / Multiplayer

**Причина отказа:** Campaigner — single-user desktop приложение. Мультиплеер требует серверной инфраструктуры, аккаунтов, синхронизации — это другой продукт. Если когда-нибудь — это будет Campaigner Server, отдельный проект.

### ❌ AI-генерация контента

**Причина отказа:** Сознательное решение автора. Worldbuilding — творческая активность; AI-генерация её обесценивает и создаёт зависимость от внешних API. Если пользователю нужен AI — пусть использует ChatGPT отдельно и копирует результат.

### ❌ Интеграции с Spotify / Discord / YouTube / Zoom

**Причина отказа:** Десктоп-приложение работает локально. Интеграции с внешними сервисами создают зависимость от их API, проблемы с авторизацией, нестабильность. Для саундбоарда достаточно локальных файлов.

### ❌ Mobile Companion App

**Причина отказа:** Другой продукт, другая платформа, другая команда (которой нет). Не в скоупе обозримого будущего.

### ❌ Узкоспециализированные RPG-фичи (Spell Slots, Feats, Sanity и т.д.)

**Причина отказа:** Campaigner системно-агностичен. Привязка к конкретной RPG-системе сужает аудиторию и создаёт ad-hoc реализации для каждой системы. Пользователи D&D могут моделировать spell slots через свободные поля.

### ❌ «Семейный кузнец / портной / пекарь / …» (35+ ролей)

**Причина отказа:** Это не 35 фич, а одна фича «связанные NPC у династии с ролями». Решается через существующий `character_relationships` + поле `role`.

### ❌ «Карты звёзд / демонов / ангелов / фей / драконов» (50+ типов карт)

**Причина отказа:** Это не разные фичи, а просто **разные пользовательские карты** в существующем домене Maps. Не требует кода.

### ❌ Отдельные домены для каждого типа преступников / магов / фанатиков

**Причина отказа:** Это значения в `factions.type` или теги, а не отдельные домены. Расщепление одной фичи на десятки — антипаттерн.

### ❌ Поля характера типа «запах / любимая еда / прощание / боевой клич»

**Причина отказа:** Это содержимое свободного описания персонажа, а не отдельные структурированные поля. Делать 50+ полей в форме создания персонажа — UX-катастрофа.

### ❌ Cloud Save / Sync

**Причина отказа:** Требует серверной инфраструктуры и аккаунтов. Бэкап решается локально (auto-save, export файлов). Если пользователь хочит облако — пусть кладёт файл проекта в Dropbox/Google Drive.
